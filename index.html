Android P SQLite 查询操作失败

标签（空格分隔）： Android SQLite

问题描述

  最近，一个发布很久且运行稳定的App突然报错，查看了错误日志，发现是一个莫名其妙的错误，解析区域范围bounds时，抛出了NumberFormatException，但是查询查询数据表，并未发现异常字符，真机调试也不会出现这种情况。我查看了一下库里面的所有数据，发现一个很奇怪的问题：
  App启动的时候，会从asset内拷贝一个数据库到SD卡，这个库里面只有两张表：tab_scope和tab_point，SQL 语句 select bounds from tab_scope 的查询结果竟然是tab_point的code字段。结果在使用Float.parseFloat()方法将文字转换为浮点数时爆出了这个异常。

  事出反常必有妖，~_~。

原因分析

我在SQLite官网找找到了这么一段话

If the last connection to a database crashed, then the first new connection to open the database will start a recovery process. An exclusive lock is held during recovery. So if a third database connection tries to jump in and query while the second connection is running recovery, the third connection will get an SQLITE_BUSY error.

翻译一下，大概意思就是讲加入最后一次的数据库连接被异常终止了，那么新打开的连接会启动一个恢复过程，在恢复期间会建立一个排它锁。如果这时候有第三个连接，第三个连接会收到一个 “SQLITE_BUSY” 错误。
不考虑锁的问题，这段话里面有个关键：”start a recovery process”。App在4.8.3版本时升级了一次数据库，tab_point就是在这个版本加进来的。那我们现在就可以设想一下这个过程了：

  • Android Pie修改了SQLite库，默认启用了wal模式;
    –在Android9.0以下系统的手机没有发现此问题
  • App退出的时候并没有主动关闭数据库连接，也就是说”last connection to a database crashed”，数据库连接异常终止了；
  • App升级后，通过拷贝的方式更新了数据库，添加了新表tab_point;
    –出问题的手机都是App半年以上未更新的
  • App再次启动，建立新的数据库连接后，SQLite恢复了数据库，结果只tab_point表格被恢复没了（经过测试，在tab_point执行查询时会抛出SQLiteException:no such table），只留下了tab_scope。不过由于数据库文件的内容已经发生了变化，原先存储tab_scope的空间，现在变为了tab_point，结果查询tab_scope里面的bounds，返回了tab_point内的code(这两个都是各自表内的第六个字段)。

说明：至于为什么还能正常执行查询语句，因为涉及的SQLite数据存储的结构、SQLite执行查询语句的过程，我尚不了解，这里不能解释。

解决方案

  1. 升级App后删除之前的wal文件；
  2. 禁用wal模式；

  3. 不在使用数据库后，主动条用数据库连接的close()方法；

    后两种方法只能预防，对于已经出现了该问题的情况，删除wal文件是最好的方法。

在Mac上搭建Https服务

生成 CA 私钥

openssl genrsa -out private/cakey.pem 2048

生成 CA 证书

openssl req -new -x509 -key private/cakey.pem -out cacert.pem -days 3650

输入如命令后，会依次让你输入国家名称，省名，城市名称，单位名称，部门名称，服务器名，电子邮件地址

生成服务器私钥

openssl genrsa -out server.key 4096

生成证书请求文件

openssl req -new -key server.key -out server.csr

同样要输入国家名称，省名，城市名称，单位名称，部门名称，服务器名，电子邮件地址等数据

###新建一个openssl.cnf配置文件

根据配置文件里面的设置，新建相应的文件夹和文件

使用自己的CA证书签发

openssl ca -in server.csr -out server.crt -cert cacert.pem -keyfile private/cakey.pem -days 3650 -config openssl.cnf -extensions SAN

注意这里的-extensions SAN 对应的配置文件中的[SAN]

生成的server.crt 即是服务器证书

问题：
(1) TXT_DB error number 2问题
srial文件里面的数 字修改为01，删除indxt.txt.attr文件，清空index.txt的内容，重新执行命令；

拷贝证书

将server.key 和 server.crt拷贝Apache 目录下

###修改apache配置
打开httpd.conf,取消一下三行的注释
LoadModule ssl_module libexec/apache2/mod_ssl.so
Include /private/etc/apache2/extra/httpd-vhosts.conf
Include /private/etc/apache2/extra/httpd-ssl.conf

打开extra/httpd-ssl.conf文件
配置以下两个参数
SSLCertificateFile “/private/etc/apache2/server.crt”
SSLCertificateKeyFile “/private/etc/apache2/server.key”

打开 extra/httpd-vhosts.conf文件，按照下面方法修改配置文件

###重启apache

sudo apachectl restart
